NATURAL HAZARD TRACKER - FEATURE REPORT

This report explains what we have implemented in the website and backend for the Natural Hazard Tracker part of the project.

1) Overall website flow
- we built an interactive map-based website where the user can click any location and analyze historical disasters near that point.
- The interface has two tabs: Cyclones and Tsunamis.
- User can select a search radius from the dropdown and run location-based analysis.
- A red click marker and a radius circle are shown on the map for clarity.
- we also added a "Clear All" function to reset layers, filters, and results.

2) Cyclone tracking features
- When the user clicks on the map in Cyclone mode, the system fetches all historical cyclones passing within the chosen radius.
- It draws cyclone tracks on the map and also shows recorded observation points (dots) along each track.
- Clicking a point opens detailed popup information:
  - timestamp (UTC),
  - latitude/longitude,
  - wind speed,
  - pressure,
  - storm nature,
  - observation number,
  - distance to land.

3) Wind intensity visualization
- we color-coded the cyclone tracks by wind intensity category.
- Not only dots, but track segments are also colored by observed wind speed.
- we added a legend on the map (N/A, TD, TS, Cat 1 to Cat 5) so interpretation is easy.
- For wind values not available, we used dark gray (close to black), as discussed.

4) Cyclone result table improvements
- The cyclone table supports sorting and filtering.
- Main fields shown:
  - Name,
  - Year,
  - Distance from clicked point,
  - Max wind speed,
  - Minimum pressure,
  - Minimum distance to land,
  - Start and End dates.
- When user clicks a cyclone row, that cyclone is highlighted on the map and others are dimmed.
- Map automatically zooms to that selected cyclone.

5) Filtering improvements (table + map sync)
- we added year-range filtering (From year - To year).
- After applying filter, the table updates as expected.
- we also implemented synchronization so only filtered cyclone tracks remain visible on the map.
- This solves the earlier issue where filtering only changed the table but not the map.

6) "No cyclone found" case handling
- If no cyclone is found within selected radius, the app now shows a proper fallback card.
- It calculates and displays the nearest historical cyclone with:
  - cyclone name,
  - year,
  - nearest distance from clicked location.

7) Distance-to-land integration
- we integrated IBTrACS distance-to-land information into the system.
- Distance-to-land is shown at:
  - each cyclone observation popup (point-wise),
  - cyclone results table (minimum distance per cyclone),
  - downloaded CSV file (minimum distance column).
- we verified data exists in database and is correctly connected.

8) CSV export enhancements
- CSV export now includes search metadata:
  - clicked latitude,
  - clicked longitude,
  - search radius,
  - result count.
- It also exports enriched cyclone fields (including min distance to land).
- we added full cyclone track geometry in WKT format (LINESTRING), so the file can be reused in GIS tools.
- Export follows currently filtered results.

9) Wind risk analysis feature (new analytical module)
- we integrated a wind-risk analysis inspired by the IBTrACS notebook logic.
- For clicked location + selected radius, system collects wind samples from nearby cyclone observations.
- Then backend performs GEV fitting (`genextreme.fit`) and computes:
  - 50-year wind speed level,
  - 100-year wind speed level.
- In results UI, I added a dedicated Wind Risk Analysis section showing:
  - sample count,
  - storm count,
  - mean and median wind,
  - 50-year and 100-year values,
  - fitted GEV parameters.
- we also added distribution visualization:
  - wind histogram,
  - fitted GEV curve,
  - marker lines for 50-year and 100-year levels.

10) Tsunami module features
- In Tsunami tab, user can similarly click on map and search historical tsunami events in radius.
- Events are plotted with severity-based marker styling.
- Tsunami table includes location, year, distance, max water height, magnitude, deaths, and damage.
- Sorting and CSV export are also available for tsunami data.

11) Backend/API work completed
- we implemented and extended FastAPI endpoints for:
  - cyclone search near clicked location,
  - cyclone track geometry,
  - cyclone observation points,
  - nearest cyclone fallback when no result,
  - wind risk analysis with GEV fitting.
- PostgreSQL + PostGIS spatial queries are used for accurate distance-based retrieval.

12) Tech stack used
- Frontend: HTML, CSS, JavaScript, Leaflet, Chart.js
- Backend: Python (FastAPI), PostgreSQL, PostGIS
- Statistical analysis: NumPy, SciPy (GEV / extreme value fitting)
- Dataset: IBTrACS cyclone dataset + tsunami dataset

Conclusion:
In this project, we moved beyond simple map plotting and implemented a full location-based disaster analysis workflow, including geospatial filtering, rich visualization, CSV export, nearest-event fallback logic, and statistical wind-risk estimation (50-year/100-year levels) for selected points.
